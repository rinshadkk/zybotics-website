What is ZybOS

An OS layer purpose-built
for the agent era

Modern AI workflows are not single models making single calls. They are networks of specialized agents — each with distinct capabilities, context windows, tool access, and failure modes — all running concurrently and passing results to one another.

ZybOS provides the infrastructure glue that makes this tractable: a typed message bus, a capability registry, a centralized LLM gateway with cost controls, a long-term Cortex memory store per agent, a workflow planner and executor with step chaining, and a fully sandboxed code runtime — all observable in real time through a live dashboard.

You bring the business logic. ZybOS handles the plumbing, the security, and the observability.

Platform Features

shipped as one system

🔀

Agent Orchestrator

FastAPI WebSocket hub for agent registration, capability discovery, heartbeat monitoring, and intelligent message routing. Agents register once and persist their UUID across reconnects — no manual re-wiring.

⚡

LLM Proxy Gateway

All agent LLM traffic flows through a single gateway. Supports Anthropic, OpenAI, and Gemini. Enforces rate limits, daily token budgets, blocked-phrase guardrails, and per-agent cost analytics — no runaway spend.

🧠

Cortex Long-term Memory

Every agent gets a persistent Markdown memory store indexed by SQLite FTS5. Facts, preferences, patterns, and instructions survive restarts and are retrievable by full-text search. Auto-compressed by LLM when entries grow.

📋

Workflow Planner

Describe a goal in natural language. The planner fetches live capability schemas from all connected agents, calls the LLM, generates a typed WorkflowPlan, and hands it directly to the executor — zero manual step wiring.

▶️

Workflow Executor

Runs plans step-by-step with SQLite persistence and {{steps[N].output.field}} template chaining so each step's output flows into the next. Live SSE progress streaming. Manual cancel at any point.

📊

Real-time Dashboard

Full control plane in one view: active agents, workflow trace with per-step status, LLM proxy stats and request inspector, Cortex memory browser, and system-wide settings — all updating live via SSE.

🌐

Browser Agent

Playwright-powered web automation agent with a ProxyProvider so all LLM calls route through the orchestrator gateway. Navigate, extract, and interact with any webpage from a capability request.

🔒

Sandboxed Code Execution

Multi-language code runner with a 9-layer security model. Kill switch, language allowlist, AST analysis, subprocess isolation, environment scrubbing, output caps, and POSIX resource limits. Unsafe modules are blocked at the import hook level — not just the OS level.

📄

Document Agent

PDF and image text extraction using pdfminer, EasyOCR, and pdf2image — all local, no third-party API. An AI-assisted extraction mode requires explicit per-call user consent before any text leaves the device. Path sandboxing enforced throughout.

Security First

Security is a
first-class principle,
not an afterthought

Every execution path in ZybOS is designed with a threat model in mind. The code execution runtime implements nine discrete, independently-enforced security layers so that a compromised or misbehaving agent cannot escalate privileges, exfiltrate environment secrets, or consume unbounded resources.

1
Kill Switch
System-wide and per-language enable/disable toggle. Pushed live to all agents via settings_push — no restart required.
2
Language Allowlist
Only explicitly permitted languages (e.g. python,javascript) can be executed. Anything outside the list is rejected before parsing.
3
AST Static Analysis
Python code is parsed into an AST before execution. Calls to exec, eval, __import__, and dangerous builtins are rejected at compile time.
4
Blocked Module List
ctypes, cffi, cython are permanently blocked — never overridable. An import hook wrapper is prepended to all submitted code before the subprocess runs.
5
Subprocess Isolation
Code runs in a child process with a scrubbed environment — all API keys and credential env vars stripped before the child starts, regardless of parent process environment.
6
Network Access Control
When code_exec_allow_network=false, socket, requests, httpx, urllib, and related modules are blocked at the import hook level — not just firewalled.
7
Execution Timeout
Configurable timeout (1–300 s). The child process is sent SIGKILL on expiry — not just a polite signal — so infinite loops cannot survive.
8
Output Cap
stdout + stderr are truncated at a configurable kilobyte limit to prevent log flooding and memory exhaustion from runaway print loops.
9
POSIX Resource Limits
RLIMIT_AS caps virtual memory and RLIMIT_CPU caps CPU seconds via preexec_fn on Linux and macOS. Applied at the OS kernel level before code starts running.

✓ Explicit AI consent gate — document AI-extraction mode returns consent_required: true before any text leaves the device.
